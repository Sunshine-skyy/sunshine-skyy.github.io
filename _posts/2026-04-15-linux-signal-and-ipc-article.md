## 前言

在 Linux 系统编程里，进程并不是孤立运行的。很多实际问题都离不开多个进程之间的协作：有的进程负责生产数据，有的进程负责消费数据；有的进程需要通知另一个进程某个事件已经发生；还有的进程需要在高效率场景下共享同一份数据区域。

围绕这些需求，Linux 提供了一整套经典的进程间通信机制，包括管道、消息队列、共享内存，以及用于异步通知的信号机制。它们看起来都在解决“进程之间如何交互”的问题，但设计目标并不相同：

- 有的偏向顺序数据流传输
- 有的偏向结构化消息传递
- 有的偏向高性能共享数据
- 有的偏向事件通知与进程控制

如果把这几个知识点分开记忆，往往会觉得函数很多、概念很散；但如果沿着“进程之间如何交换数据、如何感知事件”这条主线来理解，它们之间的关系就会清晰很多。

这一章的核心可以概括成两部分：

- 进程间通信（IPC）的四种常见方式
- 信号的产生、处理、阻塞与安装

如果只先记结论，可以先抓住下面这句话：

- 管道适合简单数据流
- 消息队列适合带类型的离散消息
- 共享内存适合高效率大数据交换
- 信号适合通知事件，不适合承载复杂数据

## 一、进程间通信的四种常见方式

Linux 中常见的进程间通信方式主要有四类：

1. 管道
2. 信号
3. 消息队列
4. 共享内存

它们最大的区别，不在于“能不能通信”，而在于“通信的数据组织方式和适用场景”不同：

| 方式 | 传递内容 | 典型特点 | 适用场景 |
| ---- | ---- | ---- | ---- |
| 管道 | 字节流 | 简单、顺序、半双工 | 父子进程、shell 命令连接 |
| 信号 | 通知事件 | 异步、轻量、不适合传复杂数据 | 中断、终止、告警、状态通知 |
| 消息队列 | 带类型的消息 | 内核维护消息链表，按消息收发 | 结构化消息传递 |
| 共享内存 | 共享一段内存区域 | 效率最高，但需要同步控制 | 大量数据交换 |

从通信效率上看，通常可以粗略理解为：

- 共享内存效率最高
- 消息队列和管道更偏向“由内核中转”
- 信号更像控制通知，而不是数据通道

## 二、管道通信

### 1. 什么是管道

管道本质上是一段由内核维护的缓冲区，一端写入，另一端读取，遵循“先进先出”的数据流动方式。

它的关键特征有：

- 数据是按字节流传输的
- 读取顺序遵循先进先出
- 管道本身不是普通磁盘文件
- 管道通常是半双工的

### 2. 无名管道

无名管道通常通过 `pipe()` 创建，更适用于有亲缘关系的进程，比如父子进程。

```c
int fd[2];
pipe(fd);
```

其中：

- `fd[0]` 通常用于读
- `fd[1]` 通常用于写

创建之后，父子进程通过 `fork()` 继承这两个文件描述符，再根据各自角色关闭不需要的一端。

一个典型流程是：

1. 父进程先调用 `pipe(fd)`
2. 父进程再调用 `fork()`
3. 父子进程分别关闭自己不用的一端
4. 一端写入，另一端读取

### 3. 为什么说管道是半双工

半双工的意思不是“不能双向通信”，而是“同一根管道在同一时刻只适合一个方向的数据流动”。

如果希望两个进程互相发送数据，常见做法有两种：

- 使用两根管道
- 直接改用其他更适合双向通信的机制

### 4. 无名管道的局限

无名管道虽然简单，但限制也明显：

- 一般只适用于有血缘关系的进程
- 数据按字节流组织，缺少消息边界
- 管道容量有限
- 默认场景下主要是单向通信

### 5. 命名管道（FIFO）

命名管道解决了“无亲缘关系进程无法直接通过无名管道通信”的问题。它在文件系统中有名字，但本质仍然是管道。

创建命名管道常用：

```c
#include <sys/types.h>
#include <sys/stat.h>

int mkfifo(const char *pathname, mode_t mode);
```

你也可以直接在 shell 中创建：

```bash
mkfifo myfifo
```

命名管道的典型特点：

- 适用于无血缘关系进程
- 需要通过路径名访问
- 仍然遵循先进先出
- 本质上仍是半双工

## 三、信号机制

### 1. 什么是信号

信号可以理解为软件层面的“异步通知”。当某种事件发生时，内核或者其他进程会向目标进程发送一个信号，提醒它进行默认处理、忽略处理，或者执行自定义处理函数。

信号最重要的几个特点是：

- 异步产生
- 本质是事件通知
- 不适合承载复杂数据
- 处理方式通常比较固定

### 2. 信号的来源

信号可以来自多个来源：

- 键盘输入，例如 `Ctrl+C`
- 硬件异常，例如除零错误
- 软件条件触发
- 一个进程主动给另一个进程发信号
- 内核在特定状态变化时发送信号

### 3. 信号的处理方式

一个信号送达进程后，通常有三种处理方式：

1. 执行默认动作
2. 忽略该信号
3. 使用用户自定义的信号处理函数捕获

常见默认动作包括：

- 终止进程
- 终止并生成 core 文件
- 暂停进程
- 恢复进程运行
- 忽略

### 4. 阻塞信号与未决信号

- 被暂时屏蔽、还没有真正递送给进程处理的信号，称为未决信号
- 一旦解除阻塞，未决信号就有机会递达目标进程

更完整的流程是：

1. 信号先被产生
2. 如果当前被阻塞，就先进入未决集合
3. 当阻塞解除后，信号才被递送
4. 递送之后再按默认、忽略或捕获规则处理

### 5. 使用 `kill()` 发送信号

`kill()` 并不意味着“杀死进程”，它真正的作用是向指定进程发送某个信号。

```c
#include <signal.h>
int kill(pid_t pid, int sig);
```

典型理解：

- `pid > 0`：发给指定进程
- `pid == 0`：发给同一进程组中的进程
- `pid < -1`：发给指定进程组
- `sig == 0`：可用于检查进程是否存在或是否有发送权限

### 6. 使用 `signal()` 安装处理函数

`signal()` 是比较早期、也比较简单的信号处理接口：

```c
#include <signal.h>
typedef void (*sighandler_t)(int);
sighandler_t signal(int signum, sighandler_t handler);
```

它适合入门理解，但在实际编程里，`sigaction()` 往往更推荐，因为行为更稳定、控制更细。

### 7. 使用 `sigaction()` 进行更完整的信号控制

函数原型：

```c
#include <signal.h>

int sigaction(int signum,
              const struct sigaction *act,
              struct sigaction *oldact);
```

如果第二个和第三个参数都为 `NULL`，这个调用可以用来检查信号编号是否合法。

`struct sigaction` 最关键的成员可以这样理解：

| 成员 | 作用 |
| ---- | ---- |
| `sa_handler` | 普通信号处理函数，只接收一个 `int` 参数 |
| `sa_sigaction` | 更高级的处理函数，可额外获取信号信息 |
| `sa_mask` | 在处理当前信号期间，临时额外屏蔽哪些信号 |
| `sa_flags` | 指定处理行为，例如是否启用 `sa_sigaction` |

当 `sa_flags` 设置为 `SA_SIGINFO` 时，通常就应该使用 `sa_sigaction` 而不是 `sa_handler`。

一个最小示例如下：

```c
void handler(int signo) {
    printf("catch signal: %d\n", signo);
}

int main(void) {
    struct sigaction act;
    act.sa_handler = handler;
    sigemptyset(&act.sa_mask);
    act.sa_flags = 0;
    sigaction(SIGINT, &act, NULL);

    while (1) {
        pause();
    }
}
```

## 四、消息队列

### 1. 消息队列的基本思想

消息队列可以理解为“由内核维护的一串消息链表”。发送方把消息按一定格式放进去，接收方再按类型或顺序取出来。

相对管道来说，消息队列更像是在传递“一个个消息对象”，而不是单纯的字节流。

### 2. 为什么需要消息队列

如果使用管道来传递消息，数据往往只是连续字节，很难天然区分“哪一段是一个完整消息”。而消息队列能够显式区分消息边界，并且可以让消息带类型。

它的典型特点：

- 消息有边界
- 可以按类型接收
- 由内核统一维护
- 不要求发送方和接收方严格同步

### 3. 常用接口

System V 消息队列常见函数包括：

- `ftok()`：根据路径和项目号生成键值
- `msgget()`：创建或获取消息队列
- `msgsnd()`：发送消息
- `msgrcv()`：接收消息
- `msgctl()`：控制或删除消息队列

```c
key_t ftok(const char *pathname, int proj_id);
```

它常用于生成共享内存、消息队列等 IPC 对象的键值。

### 4. 消息队列的优缺点

优点：

- 消息边界清晰
- 支持按类型收发
- 比管道更适合组织离散消息

缺点：

- 数据仍然需要经过内核
- 相比共享内存，效率不算最高
- 需要额外管理键值、权限和生命周期

## 五、共享内存

### 1. 为什么共享内存效率高

共享内存的思路是：让多个进程直接映射到同一块物理内存区域。这样数据不需要像管道或消息队列那样频繁经过“写入内核缓冲区 - 再复制到用户空间”的过程。

> 这样使用共享内存不涉及到内核态和用户态的切换，在很大程度上提高了数据的存取效率。

更严格地说，这句话表达的是“数据交换过程中减少了中间复制与中转开销”，所以共享内存通常被认为是最有效率的 IPC 方式之一。

### 2. 共享内存的基本使用流程

常见流程如下：

1. 使用 `ftok()` 生成 key
2. 使用 `shmget()` 创建或获取共享内存
3. 使用 `shmat()` 挂接到进程地址空间
4. 读写这块共享区域
5. 使用 `shmdt()` 脱离映射
6. 使用 `shmctl()` 管理或删除共享内存

### 3. 共享内存的优缺点

优点：

- 通信效率高
- 适合大量数据交换
- 避免频繁的数据复制

缺点：

- 共享资源带来同步问题
- 需要配合信号量、互斥机制等进行并发控制
- 生命周期管理比普通文件 I/O 更复杂

## 六、管道、消息队列、共享内存的对比

可以把三种主要数据型 IPC 方式简单对比如下：

| 机制 | 数据组织方式 | 亲缘关系要求 | 效率 | 典型问题 |
| ---- | ---- | ---- | ---- | ---- |
| 无名管道 | 字节流 | 通常需要有亲缘关系 | 中等 | 半双工、边界不明显 |
| 命名管道 | 字节流 | 不要求有亲缘关系 | 中等 | 仍然是半双工 |
| 消息队列 | 离散消息 | 无亲缘关系也可 | 中等偏上 | 内核中转、管理复杂 |
| 共享内存 | 共享内存区 | 无亲缘关系也可 | 高 | 必须自行同步 |

如果按“数据传输能力”来理解：

- 管道：最适合简单顺序数据流
- 消息队列：最适合结构化消息
- 共享内存：最适合高频、大块数据交换

如果按“控制通知能力”来理解：

- 信号：最适合做事件通知，不适合做大数据传递

---

## 总结

这一章真正要掌握的，不是单个函数原型，而是不同通信方式的设计取舍：

- 管道更轻量，适合简单串联
- 消息队列更结构化，适合传递一个个独立消息
- 共享内存效率最高，但同步控制最关键
- 信号主要承担异步通知角色

如果把这四种机制放到一起看，可以把它们理解成 Linux 提供给进程协作的四种不同工具：

- 有的偏数据流
- 有的偏消息
- 有的偏共享区域
- 有的偏控制事件

理解它们各自“最适合解决什么问题”，比死记硬背函数名更重要。
